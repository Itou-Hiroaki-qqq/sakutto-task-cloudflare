### TC-01: 単発タスク（昨日期日）が今日のリストに引継ぎ表示される

**カテゴリ:** 正常系

**前提条件:**
- 昨日の日付（例: 2026-03-20）を due_date とする単発タスクが未完了で存在する

**期待結果:**

**確認ポイント:**
- [ ] ブラウザの DevTools → Network タブで `GET /api/tasks?date=2026-03-21` のレスポンスを確認し、`is_carryover: true` のタスクが含まれていること

## 7. 発見した潜在的問題・懸念事項

以下の問題・懸念事項がコードレビュー時に発見された。

### 問題 1: キャッシュに引継ぎタスクが含まれて今日以外の日付でも汚染される可能性

**深刻度:** 中

**発生条件:**
`top/page.tsx` のプリフェッチ処理（`useEffect` の prefetch フェーズ）で今日以外の日付を fetch する際、引継ぎタスクは含まれない（`!isSameDay(date, todayJST)` の条件が API 側にある）。そのため今日以外の日付のキャッシュが汚染されることはない。ただし、今日の日付のキャッシュには引継ぎタスクが含まれる。

**懸念点:** `updateTasksCache` で今日のキャッシュを更新する際に引継ぎタスクが含まれる。そのキャッシュを再利用する際に `is_carryover` フィールドが正しく保持されているかどうかを確認が必要。

**確認方法:** TC-15 で確認。

---

### 問題 2: 引継ぎタスクの楽観的更新後にサーバーからの再取得でソートが変わる

**深刻度:** 低

**発生条件:**
引継ぎタスクにチェックを入れた後（楽観的更新）、サーバーから再取得したタスクリストとのマージ処理で、ソート順が変わる可能性がある。

**実装上の詳細:**
`handleToggleCompletion` はエラー時にサーバーから再取得する（`router.push` 後の再ロード）が、成功時はキャッシュ更新を `previousTasks` ベースで行う（top/page.tsx の 317〜325 行）。この `previousTasks` には引継ぎタスクが含まれているため、整合性はとれている。

**残懸念:** 完了トグル成功後にページが再表示されたとき（日付切替など）、サーバーから再度引継ぎタスクを取得するため表示が正しくなるはず。ただし完了直後の一瞬（APIレスポンス前）は楽観的更新でソートが適用され、完了した引継ぎタスクが完了グループに移動する。

---

### 問題 3: `getTodayJST()` の実装で `new Date()` が UTC で動作することへの依存

**深刻度:** 低（想定通りだが明示的なメモが必要）

**詳細:**
`timezone.ts` の `getTodayJST()` は `new Date().getTime()` に9時間分（`9 * 60 * 60 * 1000`）を加算してJSTを求めている。Cloudflare Workers 環境では `new Date()` は UTC を返すため、この計算は正しい。しかし、ローカル開発環境（`next dev`）では実行環境のタイムゾーンに依存する可能性がある。

**対策:** ローカルのタイムゾーンが JST（UTC+9）の場合は問題ないが、UTC や他のタイムゾーンで開発している環境では `getTodayJST()` の結果が意図しない値になる可能性がある。

**推奨:** ローカル開発時はシステムタイムゾーンを JST に設定するか、環境変数でタイムゾーンを指定する（`TZ=Asia/Tokyo`）。

---

### 問題 4: `getCarryoverTasks` 内で全タスクを取得する N+1 傾向

**深刻度:** 低（現状の規模では問題なし）

**詳細:**
`getCarryoverTasks()` は `getTasksForDate()` と同様に全タスクを取得してから日付ループを行う。今日のリクエスト時には `getTasksForDate()` と `getCarryoverTasks()` の2回全タスクを取得するため、DBへのクエリが2回実行される。タスク数が少ない個人利用アプリでは問題ないが、将来的に大量タスクを持つユーザーが増えた場合にパフォーマンスが低下する可能性がある。

**改善案:** `getTasksForDate()` と `getCarryoverTasks()` を1つの関数にまとめて、DBアクセスを1回に削減する（現状は実装の複雑さを優先して2関数に分けていると思われる）。

---

### 問題 5: `getRecurringOccurrenceDates` の `maxDays = 365` 上限と30日取得の不整合

**深刻度:** 低

**詳細:**
`getRecurringOccurrenceDates` は `maxDays = 365` を上限として出現日を計算するが、`getCarryoverTasks` は `thirtyDaysAgo` より前の出現日を除外する。このため、365日前から計算を始めても、最終的に30日以内のものだけが採用される。パフォーマンス的には冗長な計算が発生しているが、正確性は保たれている。

**改善案:** `getRecurringOccurrenceDates` に `startDate` パラメータを追加して、`thirtyDaysAgo` より前の出現日を計算しないようにする。
